# AccountCcy.com Architecture

## Purpose

This document defines the structural architecture of AccountCcy.com.

It governs the site’s page system, internal hierarchy, content roles, navigation logic, and future expansion.

The purpose of the architecture is to ensure that AccountCcy.com develops as a sovereign-grade reference asset, not as a generic finance website.

---

## Architectural Principle

Every page must strengthen the governing thesis:

**AccountCcy.com is the chain of custody for monetary truth inside enterprise finance.**

No page should exist merely to fill a sitemap.

No page should exist only because a keyword exists.

No page should exist if it weakens the asset’s authority, focus, or future acquisition value.

---

## Structural Hierarchy

The site architecture follows this hierarchy:

**Monetary Truth**  
↓  
**Chain of Custody**  
↓  
**Currency State Control Framework**  
↓  
**CCY State Chain**  
↓  
**Account Currency Control**  
↓  
**ERP / CFO / Treasury / Reporting / Audit Application**

The architecture must make this hierarchy visible to users, search engines, and future buyers.

---

## Core Page System

The initial sovereign build should include thirteen primary pages.

### 1. Homepage

Path:

`/`

Role:

The homepage establishes the asset’s supreme positioning.

It introduces AccountCcy.com as the chain of custody for monetary truth inside enterprise finance.

It should communicate:

- the governing thesis
- the hidden control problem
- the Currency State Control Framework
- the CCY State Chain
- the audience
- the asset’s strategic seriousness

The homepage must not behave as a domain sales page.

---

### 2. The Chain of Custody for Monetary Truth

Path:

`/monetary-truth-chain-of-custody/`

Role:

This is the philosophical anchor page.

It explains why monetary truth is not created automatically at payment or reporting.

It defines the concept of monetary truth custody and shows how money becomes defensible accounting reality through controlled states.

This page should become one of the asset’s highest-value conceptual pages.

---

### 3. The Currency State Control Framework

Path:

`/framework/`

Role:

This is the structural framework page.

It explains the Currency State Control Framework as original conceptual architecture.

It should define the logic of currency as a state-bearing object inside enterprise finance.

It must clearly distinguish between original framework language and established accounting or ERP terminology.

---

### 4. The CCY State Chain

Path:

`/ccy-state-chain/`

Role:

This page explains the eight states of the CCY State Chain in depth:

1. Quoted Currency
2. Transaction Currency
3. Settlement Currency
4. Account Currency
5. Ledger Currency
6. Reporting Currency
7. Consolidation Currency
8. Audit Reality

Each state should include:

- definition
- institutional meaning
- risk if misunderstood
- relationship to previous and next states
- relevance to account currency control

---

### 5. What Is Account Currency?

Path:

`/what-is-account-currency/`

Role:

This is the primary definitional SEO anchor.

It should define account currency with precision and explain how it differs from transaction currency, settlement currency, ledger currency, reporting currency, and functional currency.

It must not be generic.

It must position account currency as a control discipline.

---

### 6. Transaction Currency vs Account Currency vs Reporting Currency

Path:

`/transaction-vs-account-vs-reporting-currency/`

Role:

This is a comparison authority page.

It should explain that the same financial event can carry multiple currency identities.

It should clarify the differences between:

- transaction currency
- account currency
- reporting currency

It should connect misunderstanding these identities to reconciliation complexity, FX distortion, and reporting issues.

---

### 7. Account Currency in ERP Systems

Path:

`/account-currency-in-erp/`

Role:

This is the ERP application page.

It should explain account currency behavior and risk across enterprise financial systems without claiming official affiliation with any vendor.

It may discuss ERP contexts such as SAP, Oracle, NetSuite, Microsoft Dynamics, and similar systems in a careful, reference-oriented way.

It should serve ERP architects, implementation consultants, controllers, and finance systems professionals.

---

### 8. Functional Currency Explained

Path:

`/functional-currency-explained/`

Role:

This is the standards-grounding page.

It explains functional currency, presentation currency, reporting currency, and their relationship to account currency.

It must remain explanatory and reference-oriented.

It must not provide accounting, tax, legal, or audit advice.

---

### 9. Revaluation and Translation

Path:

`/revaluation-and-translation/`

Role:

This page explains the consequences of currency state movement.

It should cover:

- foreign currency revaluation
- currency translation
- remeasurement
- FX gains and losses
- reporting currency effects
- consolidation effects
- close-process implications

The page should connect these topics back to account currency control.

---

### 10. CFO Guide to Hidden Account Currency Risk

Path:

`/cfo-guide/`

Role:

This is the strategic value page.

It should explain why CFOs, group controllers, and finance leaders should care about account currency control.

It should elevate the asset beyond technical documentation and show how weak account currency governance can affect reporting confidence, reconciliation burden, close quality, and audit defensibility.

### 12. Currency State Diagnostic

Path:

`/currency-state-diagnostic/`

Potential maturity levels:

- Configured

`/acquisition/`

- domain

- reference architecture

## Navigation Architecture

Primary navigation should initially include:

- Doctrine
- Framework
- State Chain
- Account Currency
- CFO Guide
- Glossary
- Diagnostic

Acquisition should be visible but restrained.

It should not dominate the navigation.

---

## Internal Linking Rules

Every page should link upward to the governing thesis.

Every page should link laterally to related definitions.

Every page should link back to:

- The Chain of Custody for Monetary Truth
- Currency State Control Framework
- CCY State Chain
- What Is Account Currency?

Internal links should reinforce conceptual hierarchy, not merely SEO.

---

## URL Rules

URLs must be clean, lowercase, and stable.

Preferred format:

`/example-page/`

Avoid:

- `.html` in public URLs
- dates in permanent pages
- vague slugs
- keyword-stuffed slugs
- duplicate concepts

---

## Expansion Rules

Future pages may be added only if they strengthen one of the following:

- governing thesis
- Currency State Control Framework
- CCY State Chain
- account currency control discipline
- ERP application depth
- CFO-level value
- glossary depth
- diagnostic tool usefulness
- monetization compatibility
- acquisition value

Pages should not be created for low-quality traffic.

---

## Prohibited Architecture

The site must not develop sections such as:

- forex trading tips
- crypto speculation
- retail currency converter tools
- generic finance news
- low-grade blog posts
- affiliate product lists
- domain-for-sale landing sections
- unrelated accounting topics
- generic business content

These would dilute the asset.

## Architecture Test

Before adding any page, ask:

**Does this page strengthen AccountCcy.com as the chain of custody for monetary truth inside enterprise finance?**

If no, it should not exist.
